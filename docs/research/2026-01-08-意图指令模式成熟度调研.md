# **基于意图的结构化指令（Intent-Based Commands）在富文本编辑中的架构演进与成熟度评估报告**

## **1\. 执行摘要**

随着大型语言模型（LLM）在知识工作流中的深度渗透，富文本编辑器（Rich Text Editor, RTE）正经历一场从“文本生成辅助”向“意图驱动编辑”的根本性架构变革。传统的生成式重写模式（Generative Rewriting）虽然在创意写作中表现出色，但在处理结构化文档时面临着结构破坏、上下文丢失和计算资源浪费等严峻挑战。本报告针对“意图风格的结构化指令”（Intent-Based Commands，以下简称“选项 B”）进行了详尽的技术成熟度与可行性分析。

研究表明，选项 B 代表了 AI 编辑器的未来主流方向。该架构不再将文档视为非结构化的字符流，而是将其视为具有严格 Schema 约束的数据库。LLM 的角色从“作家”转变为“逻辑推理引擎”，负责将用户的自然语言意图转化为确定性的、结构化的操作指令（如 JSON Patch 或工具调用）。这种范式转移利用了现代 LLM 卓越的代码推理（Code Reasoning）能力，通过 Tiptap AI Toolkit、Vercel AI SDK 等中间件层，实现了对文档对象模型（DOM）的安全、精确操作。

虽然该技术尚处于早期多数（Early Majority）采用阶段，但核心组件已达到生产级成熟度。以 Qwen 2.5 Coder 为代表的开源模型在结构化指令生成上展现了极高的可靠性；Tiptap 等头部编辑器框架已推出标准化的意图执行接口；Vercel 等基础设施厂商则解决了结构化数据的流式传输难题。尽管在复杂列表操作和长文档上下文管理方面仍存在算法挑战，但“基于意图的结构化指令”已被证实是构建高可靠性 AI 编辑器的唯一可行路径。

## ---

**2\. 引言：生成式编辑的结构性危机与范式转移**

在探讨“意图风格的结构化指令”之前，必须首先剖析当前主流的“生成式重写”模式为何在复杂的富文本编辑场景中遭遇了不可逾越的瓶颈。这不仅是技术实现的差异，更是对 AI 在人机交互中角色认知的根本性分歧。

### **2.1 文本生成的幻觉与结构性熵增**

富文本编辑器（如基于 ProseMirror 或 Tiptap 构建的编辑器）中的文档并非简单的字符串，而是一个复杂的、嵌套的树状结构——文档对象模型（DOM）。一个看似简单的段落可能包含着粗体标记、超链接属性、评论元数据以及特定的排版类名。

在传统的“选项 A：生成式重写”模式中，应用程序通常提取选区的纯文本或 Markdown，将其发送给 LLM 进行处理（例如“润色这段话”），然后将生成的文本直接替换回编辑器。这种方法实际上是对文档结构的一种“有损压缩”和“再生成的猜测”。

研究资料显示，这种处理方式面临着严重的“结构性熵增”风险 2。当 LLM 生成文本时，它是一个基于概率的预测过程。虽然它擅长预测下一个单词，但它并不理解宿主编辑器的严格 Schema 约束。

* **标签丢失与属性剥离**：如果用户要求 AI 缩短一段包含特定 data-id 属性的文本，LLM 生成的新文本往往会丢失这些不可见的属性。在协同编辑环境中，这会导致评论丢失、版本追踪断裂，甚至导致应用程序崩溃。  
* **结构破坏**：在处理表格、嵌套列表或自定义组件（如“警告框”）时，LLM 经常生成无效的 HTML 结构。例如，它可能会在不允许包含块级元素的内联节点中插入 \<div\>，或者破坏表格的 \<tr\>/\<td\> 层级关系 3。这种结构的破坏不仅影响渲染，更会导致编辑器状态管理（State Management）的混乱。  
* **上下文“腐烂”（Context Rot）**：为了让 LLM 理解上下文，开发者往往需要将大量周边文本塞入提示词（Prompt）。随着文档长度增加，LLM 的注意力机制分散，导致“上下文腐烂”，即模型忽略了特定的结构指令或产生了与上下文不一致的格式 4。

### **2.2 意图驱动架构的理论模型：从 Writer 到 Operator**

为了解决上述问题，行业开始向“意图风格的结构化指令”（Option B）转移。在这个新架构中，LLM 不再直接触碰文档的“肉体”（文本内容），而是作用于文档的“神经系统”（操作指令）。

该架构的核心逻辑是：

1. **用户意图解析**：LLM 接收用户的自然语言指令（如“把第三列的数据按降序排列”）。  
2. **逻辑推理与映射**：LLM 利用其代码推理能力，分析当前文档的结构（Schema），并将自然语言转化为一个或多个标准化的 API 调用或 JSON 指令（如 table.sort({ column: 3, order: 'desc' })）。  
3. **确定性执行**：编辑器前端接收到这个结构化指令后，调用内部经过严格测试的代码逻辑（Transaction）来执行修改。

这种分离带来的优势是决定性的：

* **安全性（Schema Safety）**：由于最终修改是由编辑器内部代码执行的，因此永远不会产生无效的 DOM 结构。编辑器只会执行符合 Schema 定义的操作 3。  
* **带宽效率**：传输简短的 JSON 指令（如 {"op": "delete", "path": "/p/3"}）比传输整个重写的段落要节省大量的 Token，从而降低延迟和成本 5。  
* **逻辑复用**：编辑器的复杂逻辑（如协同编辑冲突解决、撤销重做栈管理）可以被完整保留，不需要 LLM 重新“学习”这些规则。

**3\. 核心组件分析 I：大语言模型的代码推理能力 (Code Reasoning)**

“意图风格指令”的可行性完全依赖于 LLM 是否具备足够的逻辑推理能力，以生成精确的、符合语法的结构化数据。这正是“代码推理”（Code Reasoning）能力超越单纯“文本生成”能力的用武之地。

### **3.1 为什么是代码推理模型？**

普通的大语言模型（如早期的 GPT-3）虽然能生成流畅的散文，但在处理严格的逻辑约束和格式要求时往往表现不稳定。相比之下，经过代码数据训练的模型（Code LLMs）展现出了处理意图指令的天然优势。

根据研究资料，**Qwen 2.5 Coder** 系列模型在此领域表现出了卓越的能力 6。这些模型在海量的代码库上进行了预训练，这使得它们习得了一种特殊的思维模式：

* **结构化思维**：代码本质上是高度结构化的（AST 树）。代码模型天然理解嵌套、层级和作用域，这与富文本编辑器的 DOM 树结构同构。  
* **API 依从性**：编程任务通常涉及调用特定的 API 函数。这使得代码模型非常擅长遵循“工具调用”（Tool Calling）的定义，即根据给定的函数签名（Function Signature）生成准确的参数 8。  
* **严格的格式控制**：代码模型习惯于生成语法完美的 JSON 或 YAML，因为在编程中一个括号的缺失都会导致错误。这种对符号的敏感性直接转化为生成结构化编辑指令的高可靠性。

### **3.2 性能基准与模型选择：7B 模型的“甜点位”**

在实际落地中，选择何种规模的模型至关重要。研究数据表明，**7B 参数量级的代码模型**（如 Qwen 2.5 Coder 7B Instruct）正成为端侧或私有化部署意图编辑器的“甜点位” 9。

* **指令遵循能力**：在 HumanEval 和 MBPP 等代码基准测试中，Qwen 2.5 Coder 7B 的表现甚至超过了许多更大参数量的通用模型。这意味着它能极好地理解“将所有 H2 标题降级为 H3”这类涉及逻辑遍历和条件判断的指令 7。  
* **延迟与成本**：意图编辑要求极高的响应速度。用户无法忍受为了一个简单的格式调整等待数秒。7B 模型可以在现代消费级 GPU 甚至高端 CPU 上进行本地推理，或以极低的成本在云端运行，从而实现接近实时的交互体验 11。  
* **长上下文支持**：意图分析往往需要读取大量的文档上下文。Qwen 2.5 系列支持长达 128k 的上下文窗口，这使得模型能够“看到”整个文档的结构，从而做出全局一致的编辑决策（例如“统一全文的列表样式”） 12。

### **3.3 这里的“推理”到底在推理什么？**

在“选项 B”中，LLM 的推理过程并非创作，而是翻译与规划。  
例如，用户指令是：“把这段话改成更正式的语气，并把它移到文档末尾作为总结。”  
一个具备代码推理能力的模型会进行如下思维链（Chain of Thought）：

1. **识别意图**：检测到两个意图——“风格重写”和“移动位置”。  
2. **检查约束**：读取 Schema，发现文档末尾允许插入段落。  
3. **规划步骤**：  
   * 步骤 1：调用 rewrite\_text(target=selection, tone='formal') 获取新文本。  
   * 步骤 2：调用 move\_node(source=selection, destination='document\_end')。  
   * 或者生成一个复合指令：\[{"op": "update", "id": "uuid-1", "content": "..."}, {"op": "move", "id": "uuid-1", "to": "end"}\]。

这种将模糊自然语言转化为精确逻辑步骤的过程，正是“意图风格”的核心，也是代码模型不可替代的原因。

**4\. 核心组件分析 II：编辑器侧的模式感知与执行 (Schema Awareness)**

LLM 拥有了逻辑能力，但如果它不了解编辑器的具体规则，依然会生成“幻觉”指令。因此，架构的另一端——编辑器本身，必须具备“自我描述”的能力，这就是**模式感知（Schema Awareness）**。

### **4.1 Tiptap AI Toolkit 的模式注入机制**

目前最成熟的实践来自于 **Tiptap AI Toolkit**。Tiptap 是一个基于 Headless 理念的编辑器框架，它不仅提供了编辑界面，更提供了一套让 AI 理解文档结构的协议 2。

研究资料详细揭示了 Tiptap 如何实现 Schema Awareness 3：

* **Schema 序列化**：编辑器内部通过 getHtmlSchemaAwareness 方法，将当前的文档 Schema 转换成一段人类可读（且 LLM 易读）的描述性文本。  
* **描述内容**：这段文本不仅仅列出节点名称（如 Paragraph, Heading），还包括了它们的 HTML 标签映射、允许的属性（Attributes）、嵌套规则以及对 LLM 的自然语言解释。  
  * *示例*：“CustomAlert 节点是一个用于显示警告信息的块级元素，它包含一个 type 属性，取值只能是 'info', 'warning', 或 'error'。”  
* **Prompt 注入**：这段描述被动态地拼接到发送给 LLM 的 System Prompt 中。这相当于在每次对话前，给 LLM 发放了一本“操作手册”。

### **4.2 意图指令的执行管道：Tools 与 Primitives**

当 LLM 理解了 Schema 后，它生成的指令需要被编辑器执行。Tiptap AI Toolkit 引入了\*\*Tools（工具）\*\*的概念，将编辑操作封装为可由 AI 调用的函数 13。

* **读取工具（tiptapRead / tiptapReadSelection）**：AI 可以主动请求读取文档的特定部分。这解决了上下文窗口限制的问题——AI 不必一次性吃掉整个文档，而是可以按需“翻阅”。  
* **编辑工具（tiptapEdit）**：这是最核心的工具。它接收结构化的编辑指令。资料中提到，这个工具并不只是简单的“替换”，而是支持复杂的 Patch 操作。  
* **线程管理工具（getThreads / editThreads）**：针对评论和协作场景，专门提供了管理讨论线程的工具，这在传统文本重写模式中几乎是不可能实现的（因为评论通常存储在侧边的元数据中，而非文档正文中）。

这种工具化设计使得编辑器变成了一个**Agent 环境**。LLM 不再是一个外挂的生成器，而是一个在编辑器环境中拥有“手”和“眼”的智能代理。它“看到”（Read）文档状态，根据 Schema 思考，然后“动手”（Edit）调用工具修改文档。

### **4.3 自定义节点的 AI 适配**

富文本编辑器的强大之处在于自定义节点（Custom Nodes）。例如，一个集成了 Kanban 看板的编辑器。在“选项 B”架构下，开发者需要为这个自定义节点编写 Schema 描述 15。

* **定义扩展**：开发者定义 KanbanNode。  
* **配置 AI 上下文**：开发者在 AI 扩展配置中添加 schemaAwareness 信息，告诉 AI：“这是一个看板，包含多个列，每列包含多个卡片。要移动卡片，请修改卡片的 columnId 属性。”  
* **效果**：当用户说“把‘设计’任务移到‘完成’列表”时，AI 能够精准地生成修改 columnId 的指令，而不会试图重新生成整个看板的 HTML 结构，从而避免了破坏看板内部复杂的交互逻辑。

**5\. 核心组件分析 III：结构化输出与 SDK 中间件**

连接 LLM 大脑与编辑器躯体的神经纤维，是负责传输和验证数据的 SDK 中间件。在这一领域，**Vercel AI SDK** 已经建立了事实上的行业标准，极大地降低了“意图风格指令”的实现门槛。

### **5.1 结构化数据生成的标准化：Zod Schema**

为了确保 LLM 输出的 JSON 是百分之百符合编辑器要求的，Vercel AI SDK 引入了基于 **Zod** 的 Schema 验证机制 16。

在传统的 Prompt Engineering 中，开发者往往需要乞求模型：“请只返回 JSON，不要返回其他废话。”即便如此，模型仍可能偶尔输出错误的格式。  
而在 Vercel AI SDK 的 generateObject 或 streamObject 方法中，开发者直接定义一个 Zod 对象：

TypeScript

const editorCommandSchema \= z.object({  
  toolName: z.enum(\['updateNode', 'deleteNode', 'insertNode'\]),  
  parameters: z.object({  
    nodeId: z.string(),  
    data: z.record(z.any())  
  })  
});

SDK 会将这个 Schema 转换为模型（如 OpenAI 或 Qwen）能理解的 Function Calling 定义或 JSON Mode 约束。更重要的是，SDK 在接收到模型输出后，会在客户端进行**运行时验证（Runtime Validation）**。如果模型生成的 JSON 缺少字段或类型错误，SDK 可以自动拒绝执行，甚至将错误反馈给模型要求重试。这种机制为意图编辑提供了企业级的可靠性保障 18。

### **5.2 攻克流式传输难题：Partial JSON Parsing**

富文本编辑极其讲究交互的实时性。如果用户发出指令后需要等待 3 秒钟直到整个 JSON 生成完毕才能看到结果，体验将大打折扣。然而，JSON 是一种必须闭合的数据格式，未完成的 JSON 字符串（如 {"title": "He）是无法被解析的。

研究资料指出，Vercel AI SDK 通过\*\*Partial JSON Parsing（部分 JSON 解析）\*\*技术攻克了这一难题 18。

* **原理**：SDK 内部包含一个能够在字符流到达时即时构建 JavaScript 对象的解析器。它不需要等待右大括号 }，就能解析出当前已生成的键值对。  
* **应用**：这意味着，当 AI 还在生成长文本内容（例如续写一段话）的 JSON 字段时，前端 UI 就可以利用这个“部分对象”实时地将文字流式渲染到编辑器中。  
* **混合模式**：对于纯逻辑指令（如“删除节点”），UI 会显示“思考中”状态；对于内容生成指令（如“插入文本”），UI 则可以像 ChatGPT 一样逐字显示。这种混合流式处理能力，是“意图风格”编辑器能够匹敌传统“生成式”编辑器流畅度的关键。

### **5.3 工具调用的编排与多步推理**

意图编辑往往不是单步操作。例如“先查找关于 Qwen 的资料，然后总结并插入到文档中”。这涉及到多步推理（Multi-step Reasoning）。  
Vercel AI SDK 提供了 maxSteps 配置，允许 Agent 自动进行多次工具往返（Round Trips）20。

1. Agent 调用 searchTool。  
2. SDK 执行搜索并将结果喂回 Agent。  
3. Agent 基于搜索结果调用 tiptapEdit。  
   这种自动化的编排能力，使得编辑器不仅仅是打字机，而变成了能够自主完成复杂任务的工作台。

**6\. 算法挑战：差异计算、补丁生成与流式传输**

尽管架构清晰，但在将“意图”转化为具体的“文档更新”时，仍存在显著的算法挑战。这主要体现在如何高效、准确地计算出最小化的变更集（Diff）。

### **6.1 JSON Whisperer 与 RFC 6902：全量生成 vs. 增量补丁**

学术界提出的 **JSON Whisperer** 框架为我们提供了重要的理论视角 5。该研究指出，LLM 直接修改 JSON 文档存在巨大的计算浪费。

* **全量重写**：输入 5000 Token 的文档，输出 5000 Token 的修改后文档，仅仅为了修改一个标题。  
* **Patch 生成**：输入 5000 Token 的文档，输出 50 Token 的 RFC 6902 Patch（如 \[{"op": "replace", "path": "/header", "value": "New"}\]）。

然而，让 LLM 生成 Patch 并非易事。研究发现 LLM 在处理\*\*数组索引（Array Indexing）\*\*时极易出错。

* **索引偏移问题**：如果指令是“删除第 3 项，更新第 5 项”。当第 3 项被删除后，原先的第 5 项变成了第 4 项。LLM 往往难以在生成 Patch 时动态追踪这种索引变化，导致“更新第 5 项”的操作应用到了错误的对象上。

### **6.2 解决方案：EASE 编码与唯一标识符**

为了解决索引偏移问题，JSON Whisperer 提出了 EASE（Explicitly Addressed Sequence Encoding） 编码机制 5。  
其核心思想是：去位置化（De-positioning）。

* 不要让 LLM 依赖数组下标（Index）。  
* 在发送给 LLM 的 Schema 中，为每个节点分配一个唯一的、持久的 ID（UUID）。  
* LLM 生成的指令基于 ID 进行寻址（例如 update node\_xyz），而不是基于位置（update item\[1\]）。

Tiptap 和 ProseMirror 天然支持这种模式，因为它们的内部节点模型虽然基于位置（Pos），但可以通过插件系统为节点添加 id 属性。在构建生产级意图编辑器时，**强制要求所有块级节点携带 Unique ID** 是最佳实践。这使得 LLM 的指令变成了对数据库记录的更新，彻底规避了序列操作的数学计算陷阱。

### **6.3 差异算法的选择：Myers Diff vs. Semantic Diff**

如果不使用 JSON Patch，而是让 LLM 输出一段新文本，然后由编辑器计算 Diff 呢？  
研究资料对比了不同的 Diff 算法在 AI 编辑中的表现 22：

* **字符级 Diff（Myers 算法）**：这是 Git 使用的标准算法。对于 AI 编辑来说，它太细碎了。将 "managed" 改为 "led"，Diff 显示为 \~\~manag\~\~\*\*l\*\*ed，这对用户来说难以阅读，也难以转化为语义化的 ProseMirror Step。  
* **行级 Diff**：太粗糙。段落中改了一个字，整段都会被标记为修改。  
* **语义级 Diff（Semantic Diffing）**：这是“意图编辑”的首选。它基于单词（Word）或节点（Node）进行 Diff，并引入了“语义分组”的概念。例如，如果连续多个词发生变化，它会将其视为一个整体的“重写（Rewrite）”操作，而不是零散的增删。

在 Tiptap 的实现中，往往结合了工具调用和语义 Diff。如果 AI 调用的是 updateNode，则直接应用属性变更；如果 AI 输出的是文本建议，则使用语义 Diff 算法将其转化为 ProseMirror 的 ReplaceStep，并以 Track Changes 的形式呈现。

**7\. 用户体验维度：可视化差异与交互设计**

技术架构的成功最终必须体现在用户体验（UX）上。意图编辑带来的最大 UX 挑战是：**“它到底改了什么？”**

### **7.1 从“黑盒执行”到“透明建议”**

在早期的 AI 编辑器中，用户点击“润色”，文字瞬间变化。这种“瞬移”式的体验虽然快，但缺乏透明度，用户往往因无法确知 AI 修改了哪些细节而感到不安全，甚至产生不信任感（Trust Issue） 22。

成熟的意图编辑架构必须引入**审查机制（Review Workflow）**。

* **建议模式（Suggestion Mode）**：类似于 Google Docs 的建议模式。AI 的意图不直接修改文档，而是生成“建议（Suggestion）”对象。Tiptap AI Toolkit 支持将 AI 编辑流式传输为建议，用户可以看到绿色的新增文本和红色的删除线 2。  
* **差异高亮（Diff View）**：对于大范围的结构调整（如重新排序段落），文本级的 Diff 不再适用。此时，UI 需要提供“并排对比（Side-by-Side）”或“结构化高亮”视图，明确标识出哪些节点被移动了。

### **7.2 交互的延迟管理**

尽管 7B 模型推理很快，但相比于本地代码执行，AI 仍有显著延迟。

* **乐观更新（Optimistic UI）**：对于简单的意图（如“加粗选中的字”），编辑器可以先在 UI 上模拟执行，等待 AI 确认。但对于复杂的意图，这很难实现。  
* **骨架屏与流式反馈**：利用上文提到的 Partial JSON Parsing，UI 可以在 AI 刚生成 {"action": "create\_table"... 时就在编辑器中插入一个半透明的表格骨架（Skeleton），让用户感知到“AI 正在造表”，从而缓解等待焦虑 24。

### **7.3 多模态交互：聊天 vs. 内联**

意图编辑有两种主要的交互形态：

1. **Chat Sidebar（侧边栏对话）**：用户在侧边栏与 Agent 对话。Agent 可以在侧边栏解释它的修改意图（“我帮你调整了语气...”），同时在主编辑器中高亮显示修改。这种分离通过 tiptapEdit 工具实现，是目前最稳健的模式 25。  
2. **Inline Commands（内联指令）**：用户在文档中输入 /ai 唤起指令。这种模式更流畅，但对 AI 的上下文理解能力要求更高（因为没有对话历史作为辅助）。

**8\. 成熟度评估与行业落地现状**

回到用户最关心的问题：“目前有实践论证这个是否成熟好用吗？”  
基于 2024-2026 年（假设当前时间）的行业数据和技术指标，结论如下：

### **8.1 技术成熟度曲线（Hype Cycle）位置**

“基于意图的结构化指令”目前已跨越了“创新萌芽期”，处于\*\*“早期大众（Early Majority）”\*\*采用阶段。

* **基础设施成熟**：Tiptap AI Toolkit 的推出标志着这一架构有了标准化的商业解决方案 2。Vercel AI SDK 的广泛应用证明了结构化数据流式传输的稳定性 16。  
* **模型能力达标**：Qwen 2.5 Coder 等开源模型的出现，解决了以往必须依赖 GPT-4 才能获得稳定 JSON 输出的成本瓶颈。7B 模型足以在生产环境中处理 90% 的编辑意图 7。

### **8.2 实践论证与案例**

* **Tiptap 生态系统**：Tiptap 官方宣称其 AI Toolkit 已在 Axios、GitLab、PostHog 等大型生产力工具中通过了“实战测试（Battle-tested）” 2。这些平台拥有数百万用户，证明了该架构在高并发、协同编辑环境下的稳定性。  
* **Notion AI 与类似竞品**：虽然 Notion 未开源其底层，但从其“Ask AI to Edit”功能的行为特征（精确修改属性、转换 Block 类型而不破坏内容）来看，其背后必然运行着类似的意图驱动架构，而非简单的文本重写。  
* **Zed Editor 与现代 IDE**：在代码编辑领域，Zed 等新一代编辑器已经完全采用了基于意图的 Agentic Editing。它们利用 Tree-sitter 树结构和 LSP 协议，与 Qwen/Claude 等模型配合，实现了精确的代码重构。这为富文本编辑提供了强有力的旁证 27。

### **8.3 仍存的局限性**

虽然“好用”，但并非“完美”。目前的局限性主要集中在：

1. **复杂嵌套逻辑的失败率**：面对“将表格的第三行拆分出来，作为一个新的章节标题，并把相关内容移动到该标题下”这种多步复合指令，即便是 32B 的模型也可能在 JSON 规划上出错。  
2. **长文档的上下文成本**：虽然 tiptapReadSelection 减少了 Token 消耗，但如果用户的意图依赖于文档开头定义的术语，局部上下文可能导致 AI 误解。RAG 技术虽然能缓解，但增加了系统复杂度 28。

**9\. 结论与战略建议**

结论：  
“意图风格的结构化指令”（Option B）不仅是成熟的，而且是构建现代 AI 富文本编辑器的必选项。相比于生成式重写（Option A），它在结构完整性、交互确定性和网络传输效率上具有代际优势。随着 Tiptap AI Toolkit 等中间件的普及以及 Code LLM 的端侧化，这一架构将在未来 2-3 年内彻底取代简单的文本生成 API，成为编辑器的标准配置。  
**战略建议**：

1. **拥抱 Schema Awareness**：不要试图训练 LLM 去“猜” HTML。必须在 Prompt 中显式注入经过简化的 Schema 描述，并为关键节点强制绑定 Unique ID，以配合 EASE 编码策略。  
2. **模型选型**：优先选择 **Code-Instruct** 调优的模型（如 Qwen 2.5 Coder），而非通用 Chat 模型。它们对 JSON Schema 的依从性是系统稳定性的基石。  
3. **中间件防火墙**：利用 Vercel AI SDK 或类似的 Zod 校验层作为“防火墙”。永远不要信任 LLM 的原始输出，必须在执行 Transaction 前进行严格的类型校验。  
4. **可视化优先**：在 UX 设计上，默认采用“建议模式”而非“直接应用”。信任是 AI 编辑器的核心资产，可视化 Diff 是建立信任的唯一途径。

| 对比维度 | 选项 A：生成式重写 (Generative Rewriting) | 选项 B：意图结构化指令 (Intent-Based Commands) |
| :---- | :---- | :---- |
| **工作原理** | 文本 \-\> 文本 (Text-to-Text) | 意图 \-\> 代码逻辑 \-\> JSON 指令 (Intent-to-Code) |
| **结构安全性** | **低** (易破坏 HTML/DOM 结构，丢失属性) | **极高** (由编辑器 API 保证 Schema 合法性) |
| **Token 效率** | **低** (需重新生成未修改的上下文文本) | **高** (仅传输差异指令，Payload 极小) |
| **推理能力需求** | 侧重自然语言流畅度 (Fluency) | 侧重逻辑推理与格式依从 (Reasoning & Compliance) |
| **延迟体验** | 取决于生成文本长度，往往较慢 | 取决于逻辑思考时间，通常极快 |
| **成熟度** | 传统基线，已显疲态 | **生产就绪 (Production Ready)**，行业新标准 |

综上所述，对于追求高可靠性、复杂文档处理能力的专业编辑器产品，**完全转向基于意图的架构是经过验证且风险可控的最佳实践**。
